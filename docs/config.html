Understanding Config Files & CLIs

Overview

This package uses a common function for parsing configuration files and generating command-line interfaces: The litmon.utils.cli.cli() method.

This method is used in each script file (i.e. each file in litmon.cli) with a code block that looks like:

if __name__ == '__main__':
    config: dict = cli(...)
    cls(**config)

This methods creates a command-line interface (cli) that takes two parameters:

  1. -c,--config_fname: the name of the configuration yaml file

  2. -f,--fields: the name of the fields in the file to unpack

Then, when the script containing the above code block is called from the command line, this function loads the configuration file and unpacks the listed fields as kwargs, which are used to construct cls (more details on this below).

Example

If your file configuration file looks like

nouns: {
    animal: dog,
    superhero: ironman,
}
verbs: {
    past: ran,
    present: runs,
}
adjectives: {
    color: blue,
    size: small,
}

and you use this function to create a cli that is called via

python program.py -f nouns verbs

Then this function will return a dictionary that looks like

{
    animal: dog,
    superhero: ironman,
    past: ran,
    present: runs,
}

and those variables will be unpacked to construct cls.

This setup lets you have one main configuration file that you can pick-and-choose fields from on a program-by-program basis.

Non-dict args

If any of the fields this function tries to unpack are NOT dictionaries, then the field is used as-is. E.g. if your yaml configuration file contains

fname: test.bin

and you unpack the fname variable

python program.py -f fname

Then this function will return the dictionary

{
    fname: test.bin,
}

Constructing cls

The unpacked arguments are used to construct cls in each litmon.cli file. cls is the only class in that given file.

This is the extent of the command-line interface: Each cls runs a class as a script on construction. This is object-oriented code that is used as simple python scripts. The reason behind this is that then it is easy to inherit a new subclass that customizes the implementation, while still maintaining the same simple and clean command line interface.

API Reference

litmon.utils.cli.cli(cls: str, description: str, default: list[str]) → dict[str, typing.Any]

Create CLI that loads kwargs from config files

Parameters
cls: str

Class to be constructed from configuration parameters. This is only used as a help prompt.

description: str

Description of the program, printed out if the user uses the -h flag

default: list[str]

Default fields to use from file (if not overridden by user)

Returns
dict

All parameters loaded from all supplied configuration files
